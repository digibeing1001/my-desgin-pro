## 🧠 思维框架（贯穿全流程）

> **不是每个都要用，按需选配 2-4 项。** 设计项目越复杂，用得越多。

| 框架 | 符号 | 什么时候用 | 在哪个Phase |
|------|------|-----------|------------|
| 第一性原理 | 🔬 | 用户需求模糊时，追到最根本的设计目的 | Phase 1 追问 |
| 奥卡姆剃刀 | ✂️ | 设计流程/物料列表太繁时，砍掉多余 | Phase 4 物料选择 |
| 逆向思维 | 🔄 | 执行前先想"这个设计最可能怎么翻车" | 每个 Phase 开始前 |
| 反脆弱理论 | 💪 | 出了错不光修好，还要把教训存成规则 | R-E 自进化 |
| 二八法则 | 📈 | 物料多但核心只有几个时，优先搞定关键的 | Phase 4 优先级 |
| 复利思维 | 🌱 | 设计"每用一次就比上次更好"的审美积累 | R-A 审美学习 |
| 贝叶斯思维 | 📊 | 用户每轮反馈后，调整对审美偏好的判断 | 全程 |
| **位置四问** | 📐 | 每个物料开工前确认叙事角色/观众距离/视觉温度/容量估算（融入自huashu-design） | Phase 3-4 每个物料前 |

## 🎯 模式路由（项目复杂度分级）

> **不在简单项目上做重型流程。** 根据项目规模选择合适的深度。

| 模式 | 何时用 | 做法 |
|------|--------|------|
| 🟢 **轻量级** | 单一物料（一张海报/一个Logo） | Phase 1 快速追问(3轮) → Phase 3 直接出图 → Phase 5 合规检查 |
| 🟡 **中等** | 小型品牌套件（Logo+名片+1-2物料） | Phase 1 → Phase 2(压缩) → Phase 3 → Phase 4(精选物料) → Phase 5 |
| 🔵 **深度** | 完整VI系统（多物料+多场景） | 完整6阶段，每步充分展开 |
| 🔴 **重构** | 已有品牌升级/重塑 | 深度模式 + 额外回溯：旧品牌资产盘点、迁移策略、变更影响 |

**路由判断流程：**
```
收到设计需求
    ↓
是单一物料？
    ↓ 是          ↓ 否
轻量级        是全新品牌还是升级？
                 ↓ 全新          ↓ 升级
              物料<5个？       重构模式
              ↓ 是    ↓ 否
            中等模式  深度模式
```

## ⚠️ Top 5 翻车场景（每Phase开始前快速扫描）

| # | 翻车模式 | 表现 | 怎么防 |
|---|---------|------|--------|
| 1 | **品牌名漂移** | 不同产出用了不同品牌名 | R1逐字锁定 + brand-profile.md |
| 2 | **需求理解偏差** | 做完了才发现不是用户想要的 | Phase 1 Clarify追问法（见下）+ R8逐步确认 |
| 3 | **风格分裂** | 不同物料看起来不像同一个品牌 | R3-R7锁定 + 审美档案 + 辅助图形贯穿 |
| 4 | **合规后置** | 做完才发现违法 | R9合规前置 + Phase 5双重审查 |
| 5 | **过度设计** | 物料太多、流程太长 | 奥卡姆剃刀 + 模式路由 + 二八法则 |

## ⛓️ 品牌一致性铁律（全流程强制执行）

> **这不是建议，是硬规则。违反 = 产出作废。**

品牌名确认后，**立即**在项目目录下创建 `brand-profile.md`，写入锁定项。

### 硬规则

| # | 规则 | 违反后果 |
|---|------|---------|
| R1 | **品牌名逐字锁定**：后续所有产出必须逐字引用 brand-profile.md，不得擅自替换、缩写、意译或混用 | 产出作废，重做 |
| R2 | **禁止自作主张改名**：不得将用户确认的品牌名"优化"为自认为更好的名字 | 严重违规 |
| R3 | **色彩值锁定**：所有物料必须使用 brand-profile.md 中的 HEX 值 | 产出作废 |
| **R3-C** | **Console 设计师档案优先**：如果 `.gdpro/designer-profile.json` 存在，其审美偏好覆盖默认风格建议 | 风格漂移 |
| R4 | **字体锁定**：标题/正文字体确定后，所有物料统一使用 | 产出作废 |
| R5 | **设计哲学锁定**：Phase 2 的设计哲学是全流程创作纲领，Phase 3-6 不得偏离 | 产出作废 |
| R6 | **每阶段开头读 brand-profile.md**：进入新 Phase 时必须先读取确认 | 跳过 = 违规 |
| R7 | **品牌口号锁定**：不得在不同物料中使用不同口号 | 产出作废 |
| R8 | **逐步确认，禁止跑通流程**：每个Phase/每个核心物料完成后，必须暂停等待用户确认 | 跳过确认 = 产出无效 |
| R9 | **合规审查前置**：出品每个物料前，先扫描 `references/design-compliance.md` 内化到prompt | 做完再查 = 违规产出 |

## 🖥️ Console 前端协议（R-CONSOLE）

> **Graphic Design Pro Console 是本 Skill 的远程可视化终端，不是独立应用。**
>
> 违反以下规则 = Console 与 Skill 能力割裂 = 用户获得不完整/错误的设计产出。

### 架构铁律

| # | 规则 | 违反后果 |
|---|------|---------|
| R-CONSOLE-1 | **Console 不产生设计回复**：所有设计回复必须由 Skill 生成并通过 Gateway 返回。Console 禁止本地 mock、模拟或兜底回复。 | 用户收到假数据，信任崩塌 |
| R-CONSOLE-2 | **Console 不独立执行工作流**：Phase 推进、设计哲学创建、合规审查、设计评审等全部业务逻辑由 Skill 执行。Console 只负责渲染 Skill 的输出。 | 工作流断裂，质量门禁失效 |
| R-CONSOLE-3 | **阶段推进权归 Skill**：Console 用户点击「推进」后，必须将请求转发给 Skill。Skill 检查完成度（R8）后返回确认/拒绝。Console 不得自行推进阶段。 | 阶段跳过，产出不完整 |
| R-CONSOLE-4 | **结构化输出渲染**：Skill 返回的 `structuredOutput`（合规报告/设计哲学/Moodboard/评审评分等），Console 必须渲染为对应的 UI 组件，不得只显示纯文本。 | 用户无法阅读结构化数据 |
| R-CONSOLE-5 | **Context 完整性**：Console 每次调用 Skill 时，必须传递完整的 context（设计师档案 + 知识库 + 项目资产 + 当前阶段）。禁止省略或简化。 | Skill 丢失上下文，产出漂移 |
| R-CONSOLE-6 | **双向数据同步**：Console 中的用户操作（采纳资产、修改档案、上传文件）必须实时同步到 `.gdpro/` 目录，确保 Skill 能读取最新状态。 | 数据不一致，决策基于过期信息 |
| R-CONSOLE-7 | **离线即阻塞**：未连接 Gateway/Skill 时，Console 必须显示阻塞状态，禁止以「演示模式」「离线模式」等名义提供假功能。 | 用户误以为功能可用 |
| **R-CONSOLE-8** | **自动 Gateway 连接**：从 Agent 工具启动 Console 时，`launch_console.py` 必须自动检测启动来源（环境变量 → 父进程链 → 配置探测 → 健康检查），读取对应 Agent 的 Gateway 地址与 Token，并在打开浏览器时自动注入 `?gateway=&token=&env=` URL 参数。Console 前端收到注入参数后必须立即自动发起连接，禁止要求用户手动填写 Gateway 地址。 | 用户多一步操作 = 体验断裂，尤其非技术用户 |

### 执行规则

1. Skill 启动时，检查请求来源。如果是 `source: "console"`，按 Console 协议处理
2. Skill 返回的每条消息，必须包含 `phase` 和 `phaseAction`（如有阶段变更）
3. Skill 执行完任何操作后，必须将更新的数据写回 `.gdpro/` 目录
4. Console 渲染 Skill 输出时，优先使用 `structuredOutput`，fallback 到 `text`
5. **用户输入 `design gui` / `gdpro` / `localhost:3005` 或技能被加载时**：
   - 执行 `python scripts/launch_console.py --status` 检测 Console 是否已运行
   - 如果已运行：返回 URL `http://localhost:{port}`，可用 `preview_url` 直接打开
   - 如果未运行：`Start-Process python -ArgumentList "scripts/launch_console.py --port 3005 --silent"` 后台启动 → 等待 2 秒 → 用 `preview_url` 打开 `http://localhost:3005`
   - Console 服务的是 `console/dist/` 目录下的预构建静态文件
   - 如果 `console/dist/index.html` 不存在，提示用户先 `cd console && npm run build`
   - 启动成功后，告知用户 Console URL 和关闭方式（`python scripts/launch_console.py --stop`）

| **R12** | **核心资产协议（R-BRAND-ASSET）**：涉及具体品牌时，按「Logo > 产品图 > UI截图 > 色值 > 字体」的识别度优先级采集真实资产，写入 `brand-spec.md`，所有产出必须引用真实资产路径。禁止用CSS剪影/SVG手画/通用渐变代替真实品牌资产（融入自huashu-design） | 用通用素材代替 = 品牌识别度归零，产出的是「任何品牌都长一样的通用设计」 |
| **R15** | **用户提供素材强制使用原则（R-USER-ASSET）**：用户上传/提供的任何素材（Logo/产品图/参考图/品牌手册等）必须在接收后**立即保存到 `assets/user-uploads/` 并登记到 `brand-spec.md`**；后续所有产出（生图 prompt / HTML / Canvas / 合规审查）必须优先引用该素材。禁止忽略、遗忘或擅自用 AI 重绘替代用户提供的素材。**用户提供 Logo → 立即跳过 Logo 生成步骤** | 忽略用户素材 = 品牌识别度归零 + 用户信任崩塌；擅自重绘 = 违规 |
| **R16** | **产出路径分叉规则（R-PATHWAY）**：明确区分两种独立产出路径——① **精确排版稿**（HTML/Canvas/SVG/代码生成，像素级精确，可直接交付印刷/数字端）；② **AI 概念图**（生图模型基于 prompt 重新创作，用于风格探索/氛围参考）。**禁止将精确排版稿作为 prompt 让 AI 生图模型"重新生成"以图获得"相同但更好的图"——AI 生图不是渲染引擎，无法像素级复现 HTML 布局**。如果需要"效果图"，必须先确认用户要的是「精确设计稿」还是「场景 Mockup」 | 混淆两条路径 = HTML 设计稿被 AI 重新扭曲，产出与预期严重不符 |
| **R17** | **场景化设计真实锚定原则（R-SCENE）**：涉及前台/门头/展墙/办公室等真实空间设计时，必须基于用户提供的场景照片进行。**单张照片无法判断精确绝对尺寸**，必须通过「已知尺寸参照物」或「用户手动标注」锁定真实尺寸后，方可进入设计。效果图必须使用**图生图/图像融合**技术将设计稿融入真实场景，保持原场景的光照、透视、材质一致。**禁止脱离场景照片凭空设计"通用前台"后声称是"该位置的效果图"** | 脱离真实场景 = 比例错误 + 材质冲突 + 光照虚假，效果图无法落地 |
| **R18** | **AI 生图边界原则（R-AI-BOUNDARY）**：AI 生图模型是**创意生成工具，不是精确设计工具，更不是 3D 渲染引擎**。涉及**精确排版、跨图一致性、3D 结构折叠（包装）、严格品牌色复现**的场景，禁止使用 AI 生图作为最终交付路径。这些场景必须使用确定性工具：Canvas-design（精确排版）、svg-draw（矢量）、3D 软件/专业包装工具（结构渲染）。**每次使用 AI 生图前，必须先过 `references/ai-image-limitations.md` 自检清单**。禁止以"prompt 写得不够好"为由反复尝试突破物理不可能 | 技术路径错误导致空间扭曲、一致性崩溃、无法落地，返工成本远高于使用正确工具 |
| **R13** | **事实验证先于假设（R-FACT）**：涉及具体产品/公司/事件时，第一步必须验证存在性/发布状态/版本号，写入 `product-facts.md`，禁止凭训练语料做断言（融入自huashu-design） | 事实错了，设计方向再对也是歪的 |
| **R14** | **内容诚实测试（R-HONESTY）**：每个视觉元素加入前问自己「如果去掉它，信息是否有损？」装饰性元素不加，有内在关联的元素必须加（融入自huashu-design） | 填充内容 = 稀释品牌品味，传递「不专业」信号 |
| **R19** | **交付物源文件强制规则（R-SOURCE）**：所有设计交付物必须同时提供①**可编辑源文件**（Photoshop .PSD / Illustrator .AI / .EPS / Sketch .Sketch / Figma .fig / 保留完整图层的原生格式）和②**预览/印刷文件**（PNG/JPG/PDF）。**禁止仅以位图、AI生成图或HTML截图作为唯一交付物**。源文件必须满足：图层完整未合并、文字保持可编辑（未转曲/未栅格化）、矢量路径保留（未转位图）、颜色模式正确（印刷CMYK/数字RGB）。用户明确要求"源文件"时，不得用"预览图+说明"替代 | 无源文件 = 用户无法修改/无法印刷落地/无法二次延展；违反 = 交付作废 |
| **R20** | **HTML 效果图定位规则（R-HTML）**：HTML/CSS/Canvas 生成的渲染图是**精确排版参考稿**，用于快速确认设计方向、布局、字体和间距。**HTML 截图绝对不能直接用于印刷或作为最终交付物**，因为它本质是屏幕 RGB 渲染（72–96 dpi），缺少出血线、裁切线、套印标记、颜色条等印刷必需标记，且印刷厂不接受 HTML 截图。涉及印刷的物料，必须在 HTML 确认后，在 Illustrator/Photoshop/InDesign 中**重绘**并添加完整印刷标记，转换为 CMYK/300dpi，导出 PDF/X-1a。禁止向用户交付 HTML 截图并声称"可直接印刷" | 用户拿 HTML 截图去印刷 = 色差/锯齿/缺标记/被印刷厂拒收 = 严重交付事故 |
| **R21** | **印刷标示线强制规则（R-PRINT-MARKS）**：所有涉及实体印刷的交付物（名片/海报/画册/包装刀图/招牌等），预览文件和刀图上必须包含完整的**印刷标示系统**：①出血线（3mm）、②裁切标记（Crop marks）、③套印标记（Registration marks）、④颜色条（Color bars）、⑤折叠线/模切线（包装刀图）。缺失任何一项 = 印刷厂无法准确生产 = 交付作废 | 缺标示线 = 裁切偏差/套色不准/无法校色 = 印刷废品 |
| **R-IM0** | **启动即配置（R-IM-INIT）**：本 Skill 被 `use_skill` 加载时，**立即执行**以下检测：①扫描环境变量检测已配置哪些生图 API 密钥 → ②若已配置且存在 `.gdpro/preferred-model.json`，直接采用上次选择 → ③若**无任何配置**，**必须立即暂停**并展示可用模型列表（读取 `references/image-models.md` §一），让用户选择默认模型 + 引导配置对应 API 密钥。**禁止在未确认模型的情况下直接进入设计工作流** | 未配置模型 = 到生图阶段才发现无可用 API = 流程中断/用户体验断裂 |

### 🔬 Phase 1 Clarify追问法（融入PM-Clarity）

> **不要把错误的需求设计得很漂亮。** Phase 1 追问不只是"问清楚"，要像PM一样追问到不可再分解。

**① 重述表面需求**：用结果语言重述，不只复述用户的框架

| 用户说 | 可能的真实需求 |
|--------|--------------|
| 「我要一个高端Logo」 | 「提升品牌信任感/权威性」 |
| 「设计要大气」 | 「目标场景是什么？户外招牌要远距离可辨？还是名片上要精致？」 |
| 「想要苹果那种风格」 | 「极简+大量留白+无衬线字体+产品为中心」 |

**② 识别模糊用词**：设计领域的常见模糊词：

| 模糊词 | 追问方式 |
|--------|---------|
| 大气 | 大气给谁看？什么让它看起来大气？是留白多？还是字大？ |
| 高端 | 高端的衡量标准是什么？材质？色彩？排版？ |
| 年轻 | 什么年龄段的年轻？二次元年轻还是潮流年轻？ |
| 专业 | 专业给同行看还是给客户看？什么让它看起来专业？ |
| 简约 | 简约到什么程度？极简留白还是信息精简？ |

**③ 区分目标与手段**：
- **手段伪装成目标**：「我要一个3D Logo」→ 真实目标可能是「让品牌看起来有科技感」
- **恐惧伪装成目标**：「颜色不能太素」→ 真实恐惧可能是「怕和竞品混淆」
- **跟风伪装成需求**：「别人都有品牌IP形象」→ 真实需求是什么？IP形象服务于什么？

**④ 暴露初始假设**：这个设计需求成立的前提条件是什么？
- 用户对目标受众的描述有数据支撑吗？
- 「我们的品牌调性是XX」——是经过分析的还是拍脑袋的？
- 对竞品的判断来自观察还是想象？

**⑤ 需求精确度评分**（每轮追问后自评）：

| 分数 | 含义 | 策略 |
|------|------|------|
| 0分 | 模糊、套话（"大气""高端"） | 必须追问，不推进 |
| 1分 | 有方向但粗糙 | +1轮追问补细节 |
| 2分 | 具体但缺边界 | +1轮追问补边界 |
| 3分 | 完整清晰（有受众/场景/调性/参考） | 进入下一阶段 |

> **分数 <2 时，该阶段不计入有效追问轮数。** 轻量级≥2分即可推进，深度模式需≥3分。

**⑥ 重写需求**：给出最锐利版本的真实设计需求

### 🔍 设计决策中的假设管理

> **设计决策不能把假设当事实。** 区分硬约束和软约束。

| 类型 | 定义 | 设计中的例子 |
|------|------|-------------|
| **硬约束** | 不可违反（法规/物理/已锁定品牌） | 《商标法》§10禁用标志、印刷出血3mm、品牌色HEX值 |
| **软约束** | 可挑战的惯例 | 「Logo必须圆形」「行业都用蓝色」「标题必须加粗」 |

> **绝不在没有论证的情况下将软约束呈现为不可变的。** 发现软约束时标注：「这是惯例，不是铁律。如果你愿意挑战，我们可以讨论替代方案。」

**设计假设暴露清单（Phase 2/3产出前自检）：**
- 什么被当成了"必须的"？（如"必须有渐变"）
- 哪个设计选择承载了最大权重？（如Logo的核心识别点）
- 哪个假设如果不成立，设计需要大改？（如假设受众是年轻人→实际是中年人）

## 🔧 Skill 自进化机制（R-E1 ~ R-E7）

> **用户纠正流程偏差时，必须自动迭代规则文件，不需要用户明确要求"改skill"。** 详见 `references/skill-self-evolution.md`

| # | 规则 | 违反后果 |
|---|------|---------|
| R-E1 | **纠正即触发**：用户纠正流程/工具/参数偏差 = 立即触发自进化 | 忽略纠正 = 违规 |
| R-E2 | **三步进化**：捕获纠正→诊断根因→固化到规则文件，三步不可跳过 | 跳过固化 = 违规 |
| R-E3 | **具体可执行**：固化内容必须是具体指令，禁止"注意""确保"等模糊措辞 | 模糊修正 = 无效 |
| R-E4 | **进化可追溯**：每次进化写入 SKILL.md 末尾「进化日志」+ CHANGELOG.md | 不记录 = 违规 |
| R-E5 | **重复错误升级**：同一问题纠正2次=升级规则硬度，3次=升级为铁律 | 不升级 = 违规 |
| R-E6 | **禁止"知道了但没改"**：说"记住"的同时必须已完成文件修改 | 只说不改 = 违规 |
| R-E7 | **跨场景泛化**：从具体纠正推导通用规则，覆盖同类场景 | 仅修个案 = 不充分 |

### 🔄 Reflect 7步法（融入my-skill）

当用户纠正或设计产出被否定时，自动进入反思流程：

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1. Identify | 定位根因 | 哪个Phase出了问题？是指令缺失/有歧义/参考过时/描述太窄？ |
| 2. Read | 重读原文 | 重读相关SKILL.md和phase-N.md，确认gap存在（别靠记忆） |
| 3. Impact scan | 跨文件影响 | 搜索其他phase-N.md和references/是否引用同一概念 |
| 4. Determinism | 确定性选择 | 能写成固定规则的→改SKILL.md/phase-N；需要LLM判断的→保留指令+加示例 |
| 5. Propose | 提出方案 | 改哪个文件+加什么/删什么/改什么+为什么能防止复发 |
| 6. Confirm | 等待同意 | 向用户展示方案，等明确同意后再改 |
| 7. Apply | 执行修改 | 修改文件 + 升级版本号 + 写进化日志 + 写CHANGELOG.md |

**升级警告：**
- 同一问题连续反思2次 → 停止打补丁，考虑根本方案是否有误
- 不同问题累计3+次反思 → 结构性问题，考虑是否需要拆分Phase或重组规则

### 📐 位置四问（融入huashu-design）

> **每个物料/页面/镜头开工前必答。这一步的前半段比所有设计参数更决定输出质量。**

| 问题 | 含义 | 示例 |
|------|------|------|
| **叙事角色** | 这个物料在品牌故事中扮演什么角色？ | hero（主角）/ 过渡 / 数据 / 引语 / 结尾？ |
| **观众距离** | 观众在什么距离看这个物料？ | 10cm 手机 / 1m 笔记本 / 10m 户外招牌？ |
| **视觉温度** | 这个物料要传达什么情绪？ | 安静 / 兴奋 / 冷静 / 权威 / 温柔 / 悲伤？ |
| **容量估算** | 内容塞得下吗？ | 用纸笔画3个5秒 thumbnail 估算，防溢出/防挤压 |

**执行规则**：四问答完再确定色彩/字体/布局节奏——系统服务于答案，不是先选系统再塞内容。

### 🎯 5维度专家评审框架（融入huashu-design，与design-critique互补）

> **当用户说"评审"/"好不好看"/"review"/"打分"时，或你主动想质检时，按此框架输出结构化评审报告。**
>
> **与 design-critique 的关系**：design-critique 是 11 维度的专业设计理论评审（格式塔/色彩/排版/网格等），5维度框架是**面向用户决策的评分工具**——哲学一致性/视觉层级/细节执行/功能性/创新性各 0-10 分，附带可执行的修复清单。

#### 5维度评分标准

| 维度 | 9-10分标准 | 1-2分标准 |
|------|-----------|----------|
| **哲学一致性** | 每个细节都有 Phase 2 设计哲学的依据 | 与确立的设计哲学基本无关 |
| **视觉层级** | 视线自然沿设计者意图流动，零摩擦 | 混乱，用户不知道先看哪里 |
| **细节执行** | 像素级精确，对齐/间距/颜色无瑕疵 | 粗糙，看起来像草稿 |
| **功能性** | 每个元素都服务于目标，零冗余 | 完全被装饰淹没，失去传达能力 |
| **创新性** | 在该哲学框架内找到了独特表达 | 完全是模板或素材拼凑 |

#### 场景侧重（不同物料不同重点）

| 场景 | 最重要维度 | 次重要 | 可放宽 |
|------|-----------|--------|--------|
| 海报/封面/社交媒体图 | 创新性、视觉层级 | 哲学一致性 | 功能性（单图无交互） |
| 名片/信纸/办公物料 | 细节执行、功能性 | 视觉层级 | 创新性（专业优先） |
| 包装 | 功能性、视觉层级 | 细节执行 | 创新性（准确优先） |
| PPT/品牌手册 | 视觉层级、功能性 | 细节执行 | 创新性（清晰优先） |
| Logo | 哲学一致性、创新性 | 细节执行 | 功能性（标识优先） |

#### 评审输出模板

```markdown
## 设计评审报告

**总体评分**：X.X/10 [优秀(8+)/良好(6-7.9)/需改进(4-5.9)/不合格(<4)]

**分项评分**：
- 哲学一致性：X/10 [一句话说明]
- 视觉层级：X/10 [一句话说明]
- 细节执行：X/10 [一句话说明]
- 功能性：X/10 [一句话说明]
- 创新性：X/10 [一句话说明]

### 优点（Keep）
- [具体指出做得好的地方]

### 问题（Fix）
[按严重程度排序]

**1. [问题名称]** — ⚠️致命 / ⚡重要 / 💡优化
- 当前：[描述现状]
- 问题：[为什么这是问题]
- 修复：[具体操作，含数值]

### 快速修复清单（Quick Wins）
如果只有5分钟，优先做这3件事：
- [ ] [最有影响力的修复]
- [ ] [第二重要的修复]
- [ ] [第三重要的修复]
```

**评审原则**：评审设计，不评设计师。问题描述聚焦「设计本身」，不评价设计师能力。

### 🖼️ 素材质量门槛「5-10-2-8」原则（融入huashu-design）

> **Logo 除外：有就必须用。其他素材（产品图/参考图/配图）遵循此门槛。**
>
> 原则：搜索5轮 → 找到10个候选 → 精选2个 → 每个8/10分以上。宁可少一些，也不滥竽充数。

| 维度 | 标准 | 反模式 |
|------|------|--------|
| **5 轮搜索** | 多渠道交叉搜（官网/press kit/官方社媒/YouTube截帧），不是一轮抓前2个就停 | 第一页结果直接用 |
| **10 个候选** | 至少凑10个备选才开始筛 | 只抓2个，没得选 |
| **选 2 个好的** | 从10个里精选2个作为最终素材 | 全都用 = 视觉过载+品位稀释 |
| **每个 8/10 分以上** | 不够8分**宁可不用**，用诚实placeholder或AI生成（以官方参考为基底） | 凑数7分素材进项目 |

**8/10 评分维度**：
1. **分辨率** · ≥2000px（印刷/大屏≥3000px）
2. **版权清晰度** · 官方来源 > 公共领域 > 免费素材 > 疑似盗图（0分）
3. **与品牌气质契合度** · 和 brand-profile.md 里的「气质关键词」一致
4. **光线/构图/风格一致性** · 2个素材放一起不打架
5. **独立叙事能力** · 能单独表达一个叙事角色（不是装饰）

**为什么这个门槛是铁律**：
- 滥竽充数的素材比没有更糟——污染视觉品味、传递「不专业」信号
- 消费者看作品时，每个视觉元素都在**积分或扣分**
- 7分素材 = 扣分项，不如留空或用诚实placeholder

## 关键原则

1. **先追问，后生成** — 不急于出图，先确保需求清晰（Phase 1 Clarify追问法）
2. **品牌一致性铁律** — R1-R14一旦确认，全流程锁死
3. **品牌战略先行** — 品牌类设计先定义定位和个性，再谈视觉
4. **竞品分析必做** — 品牌类设计必须先分析竞品视觉
5. **设计哲学驱动** — 每套设计都有统一的美学纲领
6. **样稿确认再扩展** — 避免方向错误导致大量返工
7. **合规内置** — 生成时即扫描（R9），不是事后补审
8. **版权红线** — 字体版权、商标法§10、AI著作权风险，零容忍
9. **风格统一** — 所有物料共享同一视觉基因
10. **审美自检** — 每个产出都要通过审美质量自检清单
11. **辅助图形不缺席** — 辅助图形是VI基础系统的必要组件
12. **色彩有比例** — 主色60%+辅助色30%+强调色10%
13. **落地闭环** — 必须提供印刷/数字检查清单和VI维护指南
14. **输出实用** — 矢量/PSD/AI/EPS可编辑源文件是交付底线，不是只能看的位图
15. **工艺极致** — 每件作品都应看起来经过无数小时精心制作
16. **禁止自由发挥** — 品牌确认的内容是铁律，不是"参考"
17. **解释为什么，不堆砌MUST** — 给理由比下命令更有效，LLM有推理能力（融入my-skill）
18. **假设必须暴露** — 设计决策中的"惯例"必须标注可挑战，不能把假设当事实（融入PM-Clarity）
19. **模式路由** — 不在简单项目上做重型流程，按复杂度选择深度（融入PM-Clarity）
20. **复利积累** — 每个项目的审美反馈都让下一个项目更好（融入my-skill）
21. **核心资产 > 规范** — Logo/产品图/UI截图的真实资产比色值字体更重要，是品牌被认出来的根基（融入huashu-design）
22. **验证省返工** — 交付前最少1分钟的验证可以省1小时的返工（融入huashu-design）
23. **宁缺毋滥** — 7分素材不如留空，滥竽充数的素材污染视觉品味（融入huashu-design）

## 🔐 渐进式披露协议（Progressive Disclosure Protocol）

> **核心思路：不一次性暴露全部信息，按「当前必要→确认后深入→高级按需」三层递进，确保每一步的决策质量。**
>
> 这是保证执行精度和出品质量的根本机制。违反本协议 = 信息过载 → 注意力稀释 → 规则漏执行 → 出品降质。

### 三层披露模型

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1 · 表面层（Surface）                                 │
│  → 只给「此刻必须知道」的信息                                 │
│  → 让用户做「方向性」决策，不做「细节性」决策                  │
│  → 禁止提前暴露下层内容                                       │
├─────────────────────────────────────────────────────────────┤
│  Layer 2 · 细节层（Detail）  ← 用户确认/执行完 Layer 1 后解锁 │
│  → 暴露「执行所需的精确参数和约束」                           │
│  → 让用户做「细节性」决策                                     │
│  → 仍禁止暴露 Layer 3                                        │
├─────────────────────────────────────────────────────────────┤
│  Layer 3 · 高级层（Advanced） ← 用户主动要求或场景触发时解锁   │
│  → 暴露「边缘情况、备选方案、深度定制选项」                    │
│  → 不影响主线流程的完整性                                     │
└─────────────────────────────────────────────────────────────┘
```

### 强制检查点（Checkpoint Gates）

> 每个 Phase 内部按 Layer 推进，**Layer 之间必须过 Gate**。

| Gate | 触发条件 | 不过 Gate 的后果 |
|------|---------|----------------|
| **G1 方向确认门** | Layer 1 展示后 | 用户未确认方向 → 禁止进入 Layer 2 |
| **G2 细节确认门** | Layer 2 展示/执行后 | 用户未确认细节 → 禁止进入 Layer 3 或下一 Phase |
| **G3 产出验收门** | 每个 Phase 最终产出后 | 用户未验收 → 禁止推进到下一 Phase |
| **G4 一致性闸门** | 进入任何 Phase 时 | 未读取 brand-profile.md 确认锁定项 → 禁止开始该 Phase |

**Gate 执行句式（强制执行）：**
```
「{当前完成的内容} 已就绪。请确认以下两点后再继续：
  1. {方向/细节} 是否满意？（满意/需修改）
  2. 是否推进到 {下一步}？（是/否）」
```

### Phase 内分层执行规则

| 规则 | 违反后果 |
|------|---------|
| **PD-1** | 进入 Phase 时，只加载该 Phase Layer 1 的内容 | 预加载全部 = 注意力稀释 |
| **PD-2** | Layer 1 执行完毕且用户明确确认后，才解锁 Layer 2 | 未确认就解锁 = 决策基础不牢 |
| **PD-3** | Layer 2 执行完毕且用户明确确认后，才解锁 Layer 3 | 同上 |
| **PD-4** | 用户说"先做出来看看"时，只执行最小可行版本（Layer 1 的极速版），而非跳过披露 | 跳过披露 = 返工风险 |
| **PD-5** | 每个 Layer 的输出必须以「可执行决策」收尾，不能以「仅供参考」收尾 | 无决策点 = 流程悬空 |
| **PD-6** | 禁止在 Layer 1 中提及 Layer 2/3 的存在（不预告未解锁内容） | 预告 = 认知负荷前置 |
| **PD-7** | 跨 Phase 时，前一 Phase 的 Layer 3 内容不自动带入下一 Phase | 自动带入 = 信息膨胀 |

### brand-profile.md 渐进式字段解锁

品牌档案字段不按 Phase 预填，而是**按 Layer 逐步解锁写入**：

| Phase | Layer 1 解锁字段 | Layer 2 解锁字段 | Layer 3 解锁字段 |
|-------|-----------------|-----------------|-----------------|
| Phase 1 | 品牌中英文名称 | 品牌口号、行业标签 | 商标注册状态、字体授权清单 |
| Phase 2 | 设计哲学名称 | 设计哲学阐述、品牌战略 | Moodboard 参考归档 |
| Phase 3 | 主色 HEX | 完整色彩系统、字体系统 | Logo 变体清单、辅助图形策略 |
| Phase 4 | 物料清单确认 | 各物料源文件路径 | 印刷工艺备注、供应商建议 |
| Phase 5 | 合规审查摘要 | 详细审查报告 | 风险等级评估、整改优先级 |
| Phase 6 | 交付物清单 | 各文件路径+格式 | 维护指南、版本管理策略 |

**字段未解锁时，brand-profile.md 中显示为 `🔒 待 Phase X Layer Y 解锁`，禁止提前填写。**

### 与渐进式加载的区别

| 维度 | 渐进式加载（原机制） | 渐进式披露（新增机制） |
|------|---------------------|----------------------|
| 解决什么问题 | 文件太大，上下文装不下 | 信息太多，决策质量下降 |
| 控制粒度 | 文件级（按需读取文件） | 指令级（按需暴露细节） |
| 用户感知 | 无感知（后台加载） | 有感知（分层确认） |
| 核心目标 | 避免上下文溢出 | 保证每一步出品精度 |

## 📐 渐进式加载原则（融入my-skill）

> **核心思路：SKILL.md是路由表+硬规则，Phase指令和参考资源按需加载。**

| 层级 | 内容 | 大小 | 何时加载 |
|------|------|------|---------|
| 元数据 | name + description | ~200字 | 始终在上下文中 |
| 主控 | SKILL.md正文（路由+铁律+规则） | ~400行 | Skill触发时 |
| Phase指令 | references/phase-N.md | 各100-200行 | 进入对应Phase时 |
| 参考资源 | references/design-*.md, brand-cases/ | 不限 | 按需读取 |

**关键实践：**
- 大参考文件(>300行)要加目录
- 品牌案例按需读取，不默认加载全部9个
- Phase指令只在进入时读取，不预加载
- **新增：Phase 指令内部按 Layer 递进披露，不一次性暴露全文件内容**

## 🎯 核心资产协议（R-BRAND-ASSET）

> **涉及具体品牌时强制执行。这是稳定性的生命线，直接决定输出质量是40分还是90分。**
> 完整 5 步流程见 `references/brand-spec.md`。
>
> 品牌的本质是「它被认出来」。按识别度排序：

| 资产类型 | 识别度贡献 | 必需性 |
|---|---|---|
| **Logo** | 最高 | **任何品牌都必须有** |
| **产品图/产品渲染图** | 极高 | **实体产品必须有** |
| **UI 截图/界面素材** | 极高 | **数字产品必须有** |
| **色值** | 中 | 辅助 |
| **字体** | 低 | 辅助 |

**执行规则**：
- 只抽色值 + 字体、不找 logo / 产品图 / UI → **违反本协议**
- 用 CSS 剪影/SVG 手画替代真实产品图 → **违反本协议**
- 找不到资产不告诉用户、硬做 → **违反本协议**
- 宁可停下问用户要素材，也不要用 generic 填充

### 5 步硬流程（摘要版）

| 步骤 | 动作 | Fallback |
|------|------|----------|
| **Step 1 · 问** | 按优先级清单逐项问用户要资产 | — |
| **Step 2 · 搜** | 官网 brand/press 页、App Store、inline CSS/SVG | — |
| **Step 3 · 下载** | Logo(SVG→inline→avatar) / 产品图(hero→press kit→AI兜底) / UI(商店截图→官网截图) | AI 生成兜底 |
| **Step 4 · 验证** | Logo(可打开+双版本+透明) / 产品图(2000px+干净背景) / UI(真实分辨率+最新版) / 色值(警惕示范品牌污染) | — |
| **Step 5 · 固化** | 写入 `brand-spec.md`，所有产出引用真实资产路径 | — |

简化速记版：

```markdown
# <Brand> · Brand Spec
> 采集日期：YYYY-MM-DD
> 资产来源：<列出下载来源>
> 资产完整度：<完整 / 部分 / 推断>

## 🎯 核心资产（一等公民）

### Logo
- 主版本：`assets/<brand>-brand/logo.svg`
- 浅底反色版：`assets/<brand>-brand/logo-white.svg`
- 使用场景：<片头/片尾/角落水印/全局>
- 禁用变形：<不能拉伸/改色/加描边>

### 产品图（实体产品必填）
- 主视角：`assets/<brand>-brand/product-hero.png`（2000×1500）
- 细节图：`assets/<brand>-brand/product-detail-1.png`
- 场景图：`assets/<brand>-brand/product-scene.png`

### UI 截图（数字产品必填）
- 主页：`assets/<brand>-brand/ui-home.png`
- 核心功能：`assets/<brand>-brand/ui-feature-<name>.png`

## 🎨 辅助资产

### 色板
- Primary: #XXXXXX <来源标注>
- Background: #XXXXXX
- Ink: #XXXXXX
- Accent: #XXXXXX
- 禁用色: <品牌明确不用的色系>

### 字型
- Display: <font stack>
- Body: <font stack>

### 气质关键词
- <3-5 个形容词>

### 禁区
- <明确不能做的>
```

**写完 spec 后的执行纪律（硬要求）**：
- 所有产出必须**引用** `brand-spec.md` 里的资产文件路径，不允许用 CSS 剪影/SVG 手画代替
- Logo 作为 `<img>` 引用真实文件，不重画
- 产品图作为 `<img>` 引用真实文件，不用 CSS 剪影代替
- 这让品牌一致性从「靠自觉」变成「靠结构」——想临时加色要先改 spec

### 全流程失败的兜底

| 缺失 | 处理 |
|---|---|
| **Logo 完全找不到** | **停下问用户**，不要硬做 |
| **产品图（实体产品）找不到** | 优先 AI 生成（以官方参考图为基底）→ 次选向用户索取 → 最后诚实 placeholder |
| **UI 截图（数字产品）找不到** | 向用户索取自己账号的截屏 → 官方演示视频截帧 |
| **色值完全找不到** | 按设计方向顾问模式，推荐 3 个方向并标注 assumption |

**协议代价 vs 不做代价**：

| 场景 | 时间 |
|---|---|
| 正确走完协议 | 下载 logo 5 min + 下载产品图 10 min + 提取色值 5 min + 写 spec 10 min = **30 分钟** |
| 不做协议的代价 | 做出没识别度的通用设计 → 用户返工 1-2 小时，甚至重做 |
