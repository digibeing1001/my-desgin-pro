# Skill 流程自进化机制（Skill Self-Evolution）

> 硬规则文件。与 SKILL.md 同级约束力。违反 = 机制失效。
>
> **核心目的**：当用户在使用 skill 过程中发现流程偏差并纠正时，skill 必须自动将纠正内容固化到规则文件中，实现自我迭代优化，而不是每次都要用户明确要求"你去改一下 skill"。

---

## 1. 核心问题

VI 设计 skill 在运行过程中，会出现各种流程偏差：

- **工具选错**：比如内置 image_gen 超时，临时切换到豆包 API，但 skill 默认没写清豆包是主力工具
- **参数错误**：比如豆包 API 的 size 参数不支持 1024x1024，但 skill 没标注
- **流程遗漏**：比如生图前没读取 brand-profile.md，导致品牌名不一致
- **优先级错乱**：比如先用内置工具失败才用主力工具，而不是直接用主力工具
- **产出偏差**：比如自作主张改品牌名、换颜色、用不同字体

**用户每次纠正这些偏差，都是在教 skill 怎么做对。但如果纠正完了就完了，下次同样的问题还会犯。**

**本机制要解决的核心问题：用户每纠正一次，skill 就进化一次，同类问题不再重犯。**

---

## 2. 硬规则体系

### R-E1: 纠正即触发（用户纠正 = 进化信号）

**规则**：当用户对 skill 的流程、工具选择、参数使用、产出质量提出纠正时，**立即触发自进化流程**，不需要用户说"你去改一下 skill"。

**触发信号识别**：

| 用户表述 | 触发类型 | 示例 |
|----------|---------|------|
| "这里应该用 X 而不是 Y" | 工具/流程纠正 | "以后都用豆包 API" |
| "你搞错了/不对" | 产出偏差纠正 | "品牌名是超级汉堡不是 SUPER FIT" |
| "这个步骤漏了" | 流程遗漏纠正 | "你应该先读 brand-profile.md" |
| "参数不对" | 参数纠正 | "豆包不支持 1024x1024" |
| "每次都要我提醒你" | 重复错误纠正 | "品牌名一致性不是建议，是硬规则" |
| "你要想办法设置规则" | 机制缺失纠正 | "你要加死规则保证一致性" |

**不触发的场景**：
- 用户只是表达偏好（如"我喜欢绿色"）→ 走审美学习机制
- 用户要求修改设计内容（如"Logo 再大一点"）→ 走正常设计修改流程

### R-E2: 三步进化流程（捕获→诊断→固化）

**规则**：每次触发自进化时，必须执行以下三步，不可跳过任何一步。

#### 步骤1：捕获纠正

记录用户纠正的**原文**和**上下文**：

```
📋 进化捕获
━━━━━━━━━━━━━━━━━━━
触发时间：[当前时间]
用户原文："[用户纠正的原话]"
上下文：[当时在做什么、出了什么问题]
纠正类型：[工具/流程/参数/产出/机制]
```

#### 步骤2：诊断根因

分析**为什么会出现这个偏差**，找到根因而非表面问题：

```
🔍 根因诊断
━━━━━━━━━━━━━━━━━━━
表面问题：[用户看到的问题]
根因分析：[为什么会出这个问题]
  - SKILL.md 是否缺规则？→ [缺什么]
  - SKILL.md 规则不够硬？→ [哪条规则不够明确]
  - SKILL.md 流程有漏洞？→ [哪个步骤可以绕过约束]
  - 参考文件信息缺失？→ [缺什么信息]
  - 工具优先级未定义？→ [哪个工具应该是默认]
影响范围：[这个问题可能还影响哪些场景]
```

#### 步骤3：固化到规则文件

将修正**写入对应的规则文件**，而不是只在当前对话中修正：

| 修正类型 | 固化位置 | 修正方式 |
|----------|---------|---------|
| 工具优先级变更 | SKILL.md 对应 Phase | 修改默认工具说明 |
| 参数约束 | SKILL.md API 部分 | 追加约束说明 |
| 流程步骤遗漏 | SKILL.md 对应 Phase | 追加强制步骤 |
| 新增硬规则 | SKILL.md 关键原则 / 品牌一致性铁律 | 追加硬规则 |
| 新增防错机制 | SKILL.md 或新建 references/ 文件 | 创建新机制 |
| 踩坑经验 | SKILL.md 末尾「踩坑经验」区域 | 追加经验条目 |

**固化后必须验证**：
1. 重新读取修改后的文件，确认修改已写入
2. 检查修改是否与其他规则矛盾
3. 输出进化确认

### R-E3: 固化必须具体可执行（禁止模糊修正）

**规则**：固化到规则文件中的修正，必须是**具体、可执行、可验证**的指令，禁止模糊的"注意"/"小心"/"确保"。

**❌ 错误示范**：
```
- 注意使用正确的品牌名
- 确保先用豆包API
- 小心参数格式
```

**✅ 正确示范**：
```
- 品牌名必须逐字引用 brand-profile.md，不得替换/缩写/意译（R1铁律）
- 生图调用优先使用豆包即梦API（doubao-seedream-5-0-260128），内置image_gen仅作备用
- 豆包API的size参数只接受"2K"/"4K"或像素值（最小≈1920x1920），不支持1024x1024
```

### R-E4: 进化记录可追溯

**规则**：每次自进化必须记录到 SKILL.md 末尾的「进化日志」区域，形成完整的进化历史。

**进化日志格式**：

```markdown
### E-001: [进化标题]

| 字段 | 内容 |
|------|------|
| 日期 | 2026-04-20 |
| 触发原文 | "以后都用豆包 API" |
| 根因 | SKILL.md 未指定默认生图工具优先级，内置image_gen超时后才切换豆包 |
| 修正位置 | SKILL.md Phase 3「豆包即梦API调用」章节 |
| 修正内容 | 明确豆包API为默认生图工具，内置image_gen为备用 |
| 影响范围 | 所有需要AI生图的Phase（3/4/5） |
| 验证状态 | ✅ 已读取确认 |
```

### R-E5: 重复错误升级（同一问题纠正2次 = 规则升级）

**规则**：如果同一类问题被用户纠正2次及以上，说明现有规则力度不够，必须**升级规则硬度**。

**升级路径**：

| 纠正次数 | 升级动作 |
|----------|---------|
| 第1次 | 追加规则说明 |
| 第2次 | 升级为硬规则（加入⚠️标记和违反后果） |
| 第3次 | 升级为铁律（加入R编号，与R1-R7同级） |
| 第4次+ | 重构机制（创建新参考文件/防错流程） |

**示例**：
- 第1次用户纠正"品牌名不一致" → 追加"注意品牌名一致"
- 第2次用户纠正"又品牌名不一致了" → 升级为硬规则 R2"禁止自作主张改名"
- 第3次用户纠正"还是要靠规则锁死" → 创建 brand-profile.md 锁定机制 + R1-R7 铁律体系

### R-E6: 禁止"知道了但没改"

**规则**：AI 不得在对话中说"我记住了"/"以后注意"但实际没修改任何规则文件。

**检查标准**：如果对话中说了"我以后会注意"，但 SKILL.md / references/ 文件没有任何实际修改，就是违规。

**唯一合规路径**：说"我记住了"的同时，**必须已经完成了步骤3的固化操作**。

### R-E7: 跨场景泛化（从具体纠正推导通用规则）

**规则**：用户的纠正往往是针对具体场景的，skill 必须将具体纠正**泛化为通用规则**，覆盖同类场景。

**泛化方法**：

| 用户纠正（具体） | 泛化后的通用规则 |
|-------------------|-----------------|
| "品牌名应该是超级汉堡不是 SUPER FIT" | 所有品牌确认的内容不得擅自替换/缩写/意译（R1铁律） |
| "以后都用豆包 API" | 生图默认使用豆包即梦API，其他工具为备用 |
| "参数不对，豆包不支持1024x1024" | 调用任何API前必须先确认参数约束 |
| "你要设死规则" | 所有约束必须写成硬规则+违反后果，不写模糊建议 |

## 3. 自进化工作流

```
用户纠正偏差
      ↓
  ┌──────────────────────┐
  │ R-E1: 识别纠正信号     │
  │ 判断是否触发自进化      │
  └───────┬──────────────┘
          ↓ 是
  ┌──────────────────────┐
  │ R-E2-Step1: 捕获纠正   │
  │ 记录原文+上下文        │
  └───────┬──────────────┘
          ↓
  ┌──────────────────────┐
  │ R-E2-Step2: 诊断根因   │
  │ 找到规则文件的缺陷      │
  └───────┬──────────────┘
          ↓
  ┌──────────────────────┐
  │ R-E2-Step3: 固化修正   │
  │ 写入SKILL.md/refs文件  │
  │ R-E3: 具体可执行       │
  │ R-E7: 泛化为通用规则    │
  └───────┬──────────────┘
          ↓
  ┌──────────────────────┐
  │ 验证：重新读取确认      │
  │ 检查是否与其他规则矛盾   │
  └───────┬──────────────┘
          ↓
  ┌──────────────────────┐
  │ R-E4: 写入进化日志     │
  │ R-E5: 检查重复错误升级  │
  └───────┬──────────────┘
          ↓
  ┌──────────────────────┐
  │ 输出进化确认           │
  │ - 改了哪个文件         │
  │ - 加了什么规则         │
  │ - 下次不会重犯        │
  └──────────────────────┘
```

## 4. 与其他机制的关系

| 机制 | 管什么 | 触发条件 |
|------|--------|---------|
| **品牌一致性铁律（R1-R7）** | 产出必须与锁定项一致 | 每个物料产出前 |
| **Skill流程自进化（R-E1~R-E7）** | 规则文件本身的迭代优化 | 用户纠正流程偏差时 |
| **审美学习与风格固化** | 学习用户审美偏好 | 用户表达审美偏好/纠正审美选择时 |
| **设计修改管理** | 设计修改有纪律可追溯 | 用户要求修改设计内容时 |
> 见 SKILL.md 渐进式披露协议（PD-1~PD-7）+ R8 逐步确认规则 + 各 Phase Gate 系统

**核心关系**：
- 品牌一致性铁律是**产出约束**（做什么对）
- Skill自进化是**规则迭代**（怎么防止做错）
- 审美学习是**偏好固化**（怎么做更好）
- 设计迭代支持是**修改管理**（怎么安全地改）

## 5. 已发生过的进化案例

### E-001: 品牌名一致性漂移

| 字段 | 内容 |
|------|------|
| 日期 | 2026-04-20 |
| 触发原文 | "前面是 Super fit 后面又是 Super burger，这个 skill 要解决前后一致性的问题" |
| 根因 | SKILL.md 没有品牌名锁定机制，AI 自作主张将"超级汉堡"改名为"SUPER FIT" |
| 修正位置 | SKILL.md 关键原则 + Phase 1 品牌名固化 + 新建 brand-profile.md |
| 修正内容 | R1-R7 铁律体系 + brand-profile.md 锁定机制 + 每阶段强制读取 + 禁止用法黑名单 |
| 影响范围 | 全流程所有产出 |
| 验证状态 | ✅ 已验证 |

### E-002: 硬规则约束不足

| 字段 | 内容 |
|------|------|
| 日期 | 2026-04-20 |
| 触发原文 | "一定要想办法设置规则，尤其是死规则、硬规则，来保持前后品牌内容、视觉设计的一致性" |
| 根因 | 规则以建议形式存在（"注意""确保"），缺乏违反后果和防错机制 |
| 修正位置 | SKILL.md 关键原则第16条 + 品牌一致性铁律章节 |
| 修正内容 | 原则16"禁止自由发挥"+ 铁律R1-R7全部加上违反后果 + 一致性自检清单 |
| 影响范围 | 全流程 |
| 验证状态 | ✅ 已验证 |

### E-005: 流程跑通+合规后置+包装缺刀图+招牌中文偏小

| 字段 | 内容 |
|------|------|
| 日期 | 2026-04-20 |
| 触发原文 | "你把logo做完以后，应该先问用户是否满意，需不需要修改，然后再进入下一步。每一步完成之后，都要去询问用户。关于物料设计的部分，像是包装盒的设计，不是应该同时给出刀图？还有你的审查可以前置。招牌中文字体要大于英文字体才行？" |

| 修正内容 | R8逐步确认铁律 + R9合规前置 + R10包装刀图 + 自检增加3项 |

| 验证状态 | ✅ 已验证 |

### E-006: 包装物料缺成品效果图

|------|------|

| 验证状态 | ✅ 已验证 |
